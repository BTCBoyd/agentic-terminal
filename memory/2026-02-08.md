## Language Detection Bug Fix (16:28 EST)

**Issue reported:** Boyd's wife tested web chat in Spanish. Initial response came in Portuguese, then after asking for Spanish, got English summary with Spanish full explanation.

**Root cause identified (2 bugs):**

**Bug #1: Summary generation ignored language**
- Two-pass system: Full response got language param ✅, summary did NOT ❌
- Line 628-633: `callAnthropicDirect()` used hardcoded English system prompt
- Result: Spanish full response + English summary (inconsistent UX)

**Bug #2: Language detection confused Spanish/Portuguese**
- Many overlapping words: "que", "para", "está" exist in both languages
- Function checked Spanish patterns first → Portuguese messages could match Spanish
- Scoring approach needed to prioritize distinctive patterns

**Fix applied:**
1. **Summary respects language**: Added language instruction to summary system prompt + prompt requirements
2. **Improved detection**: Score-based approach using distinctive patterns first (Portuguese: ã/õ/ç/não/você, Spanish: ¿/¡/qué/cómo), fallback to common words if tied

## Inactivity Auto-Clear Implementation (16:40 EST)

**Priority: HIGH** - Token cost optimization to prevent conversation history accumulation

**Problem:** Storing full conversation history in localStorage meant sending 15,000+ tokens per query for returning users with long conversations. With two-pass system (full response + summary generation), this was extremely expensive.

**Solution Implemented:**

**1. Inactivity Tracking:**
- `lastActivityTime` tracked on every user message and assistant response
- Checked every 60 seconds via `setInterval`
- Timeout threshold: 45 minutes

**2. Auto-Clear Behavior (clearChatDueToInactivity):**
- Clears entire conversation history from UI (removes all message DOM elements)
- Clears localStorage (`maxi_messages`, `maxi_count`)
- Resets to fresh welcome state
- No conversation context sent to Claude on next query

**3. User Notification:**
- Subtle gray notification: "Previous conversation cleared after inactivity to ensure fresh context. Your questions are never stored beyond active sessions."
- Auto-dismisses after 5 seconds
- Fade-in/fade-out animation
- Positioned at top of messages container

**4. Backup for Future Restore (Phase 2 ready):**
- If conversation had 3+ exchanges, stores backup for 10 minutes
- `conversationBackup` variable holds: history, count, timestamp
- Future enhancement: "Restore Previous Conversation" button

**5. Activity Updates:**
- `updateActivity()` called in two places:
  - Start of `sendMessage()` (user sends message)
  - After successful assistant response (activity = bidirectional)

**Cost Impact:**
- **Before**: 10-message conversation = ~15,000 tokens per subsequent query
- **After**: Fresh query = ~1,500-3,000 tokens (just question + system prompts)
- **Estimated savings: 80-90%** reduction in API costs for returning users

**Technical Implementation:**
- `INACTIVITY_TIMEOUT`: 45 * 60 * 1000 (45 minutes in ms)
- `BACKUP_RETENTION`: 10 * 60 * 1000 (10 minutes in ms)
- `startInactivityMonitor()`: Interval checks every 60 seconds
- `clearChat()`: Removes messages, resets state, clears storage, shows welcome
- CSS: `.inactivity-notification` with fade animations
